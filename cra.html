CRA Reward Mechanism

June 2025

Abstract

This document outlines a robust and dynamic reward mechanism intended to incentivize licensed data providers—Consumption Reflection Agents (CRAs)—for timely and accurate submission of aggregated acts of consumption data, termed Consumption Units (CUs). The reward is dispensed in the protocol's native cryptocurrency token, coen, through synchronized global epochs lasting precisely 24 hours.

To guarantee continuous and stable participation from CRAs worldwide, the mechanism dynamically adjusts the reward pool each epoch based on two critical parameters:

  1. CRA Participation Trends: On-chain usage metrics indicating changes in CRA activity levels.
  2. Fiat Price Volatility: Real-time fiat price of coen, sourced from trusted external oracles.

The mechanism operates under a strict emission cap parameter θ, limiting CRA reward allocations within a controlled fraction of total token minting per epoch, safeguarding monetary integrity and preventing uncontrolled inflation. It means that under some conditions reward pool may be capped that may lead to reduction of CRA rewards that may force less efficient CRAs to temporarily or permanently suspend their operation and reduce volume of submitted CUs. That is essentially a network self regulation and balancing approach

Robust anti-manipulation protocols are embedded to mitigate potential gaming behaviors and secure oracle integrations, ensuring fairness, predictability, and protocol sustainability.

The protocol's primary focus lies in the generation and validation of Consumption Units. It operates under the assumption that a substantial number of entities will align with the functional criteria of the Consumption Reflection Agent (CRA), each identifying their own strategic incentives for participation. Notably, the CRA is conceptualized as a functional role rather than a fixed institutional actor.

Glossary

Term Definition
Consumption Reflection Agent (CRA) is a regulated entity responsible for retrieving, verifying, and posting transaction data. CRAs generate Consumption Units by aggregating a Consumption Records and confirming their legitimacy. They act as the bridge between traditional financial systems and the network's on-chain value creation.
Consumption Unit (CU) is an entity representing a batch of Consumption Records aggregated daily per bank account. Each Consumption Unit qualifies as a whole for Tribute issuance and serves as the basis for Tribute allocation.
Coen is a digital asset on Network whose value directly reflects Users' economic activities. Native Coins are minted strictly according to Emission Limit rules and are tradable freely on Exchanges.
Epoch (24h) is a fixed, continuous interval on the Network consisting of a defined number of consecutive blocks, aligned with the Worldwide Day—a 48-hour period covering all global time zones. Epochs structure Network operations, including Validator selection, Tribute allocation, and protocol updates, providing regular checkpoints for consensus and governance activities on a globally synchronised schedule.
Reward Pool (Pₙ) Total Coen allocated for CRA rewards in epoch n
MaxEmissionₙ Upper limit on newly minted Coen per epoch n.
Emission Cap (θ) Fraction of MaxEmissionₙ allocated for CRA rewards

System Logic

1. Lifecycle Detailed Overview

Each epoch (24 hours) consists of three clearly defined phases:

Initiation

At epoch onset:

  • The protocol fetches fiat exchange rates via decentralized oracles.
  • It calculates reward projections based on previous participation trends and fiat stability targets.
  • Initial reward pool P_n is set, respecting the emission cap. Emission cap will be defined in a subsequent phase and calibrated based on network maturity and ecosystem needs.

CU Submission Phase

During the epoch:

  • CRAs aggregate daily transactions into CUs.
  • CUs must contain cryptographic hashes ensuring uniqueness and integrity.
  • CUs are recorded on-chain (L2) with metadata. CRA Consumption Unit generation

Epoch Conclusion and Settlement

At epoch conclusion:

  • Protocol verifies CU validity and uniqueness.
  • Final reward pool P_n is distributed proportionally among CRAs based on the number of valid submissions:
Reward for CRA_i = (u_i / U_n) × P_n

where:

  • u_i is the number of valid submissions from CRA i.
  • U_n is total valid submissions in epoch n.
# [Reward Distribution]
Reward_i = (u_i / U_n) * P_n
# u_i   = valid CUs submitted by CRA i
# U_n   = total valid CUs submitted by all CRAs in epoch n
# P_n   = final coen reward pool for epoch n

2. Reward Pool Calculation

Reward pool determination integrates behavioral and market data to optimize CRA compensation dynamically.

2.1. Input Metrics and Rationale

The pool uses two primary metrics:

CRA Participation Rate:

Tracks growth or decline in submission volumes, stabilizing incentives against participation variability using exponential moving average (EMA).

Fiat Stability Target:

Ensures rewards remain predictable in fiat-equivalent terms despite crypto price volatility.

2.2. Detailed Computational Formulas

Participation-based Adjustment:

Calculate participation ratio (r_{n-1}):

r_{n-1} = U_{n-1} / EMA_{n-2}

Where:

  • U_{n-1}: Actual submissions previous epoch.
  • EMA_{n-2}: Historical participation baseline from preceding epochs.
r_n-1 = U_n-1 / EMA_n-2
# r_n-1 = participation ratio
# U_n-1 = actual CU submissions in previous epoch
# EMA_n-2 = historical average CU submissions

Adjust pool size accordingly:

P_usage = P_{n-1} × [1 + α (r_{n-1} - 1)]
  • α: responsiveness factor (recommended range 0.05–0.2).
P_usage = P_n-1 * [1 + α * (r_n-1 - 1)]
# P_n-1 = previous epoch reward pool
# α     = responsiveness factor (e.g., 0.1)
  • If CRA activity increased compared to the past (i.e., r_{n-1} > 1), then P_usage grows.
  • If CRA activity decreased (r_{n-1} < 1), then the reward pool shrinks.
  • If participation is stable (r_{n-1} = 1), then the reward pool stays the same as before.

Example:

  • P_{n-1} = 10,000
  • α = 0.1
  • r_{n-1} = 1.2 (20% more CRA submissions than usual)

Then:

P_usage = 10,000 × [1 + 0.1 × (1.2 − 1)] = 10,000 × 1.02 = 10,200 coen

So the reward pool grows by 2%.

Fiat-based Stability Adjustment:

Maintain a targeted fiat-equivalent per CU:

P_fiat = (T_fiat × Û_n) / Price_n
  • T_fiat: Fiat target per CU (set by governance, e.g., $1.50 per CU).
  • Û_n: Predicted CU submissions (EMA-based forecast).
  • Price_n: Fiat-to-coen exchange rate.
# [Fiat-Based Pool Adjustment]
P_fiat = (T_fiat * Ũ_n) / Price_n
# T_fiat  = fiat target reward per CU (e.g., 1.50 USD)
# Ũ_n     = forecasted CU submissions (via EMA)
# Price_n = coen to fiat exchange rate

Example:

  • T_fiat = 1.50 USD per CU
  • Û_n = 5,000 CUs (forecast)
  • Price_n = 0.75 USD per coen

Then:

P_fiat = (1.50 × 5,000) / 0.75 = 7,500 / 0.75 = 10,000 coen

So, you need to allocate 10,000 coen to the reward pool to ensure that CRAs are paid $1.50 worth of value per CU at the current token price.

There are several adjustments expecting for this Section.

  • CU Pricing Formula: If the CRA exhibits growth in CU volume, the price per CU should decrease.
  • Volume Adjustment: The greater the number of CUs allocated, the lower the price per CU.
  • Exchange Rate Adjustment: If the exchange rate increases, CU allocation should be reduced accordingly.

Final Reward Pool Determination:

Combine projections with emission cap enforcement:

P_n = min(θ · MaxEmission_n, max(P_usage, P_fiat)) 
# [Final Reward Pool Calculation]
P_n = min(θ * MaxEmission_n, max(P_usage, P_fiat))
# θ = emission cap fraction (e.g., 0.08)
# MaxEmission_n = max mintable coen this epoch

Step-by-step:

1. Take the higher of:

  • P_usage (activity-driven)
  • P_fiat (fiat-equivalent driven)

2. Enforce hard cap:

  • Compare that number to θ · MaxEmission_n → Only allow rewards up to the emission cap.

3. Final reward pool P_n is the smaller of the two. The smaller value between:

  • The suggested reward pool (from Step 1), and
  • The hard cap value (from Step 2)

3. Anti-Manipulation Mechanisms

Minimum Activity Threshold

  • Epoch reward pools may be partially or fully withheld if total CU submissions fall below a governance-defined minimum (minUnits), preventing strategic withholding to increase subsequent pools.

Oracle Attack Mitigation

  • Utilize decentralized multi-oracle median pricing.
  • Reject abnormal price deviations exceeding predefined thresholds.
  • Automatic fallback to participation-based rewards if oracle data becomes compromised.

CRA Bonding (Optional Security Layer)

  • Optional coen staking by CRAs, creating direct economic accountability.
  • Stake forfeiture (slashing) enforced for fraudulent or negligible participation.

Threshold-Based CU Submission Limits

To maintain healthy protocol operations and guard against outliers, the network dynamically enforces minimum and maximum CU submission thresholds per epoch. These are computed based on the Exponential Moving Average (EMA) of CU submissions over a defined recent window (e.g., last 5 epochs).

Let:

  • EMA_n: 5-epoch exponential moving average of total CU submissions
  • T_min = β_min · EMA_n: minimum threshold
  • T_max = β_max · EMA_n: maximum threshold

Where:

  • β_min ∈ [0.5, 0.8] (e.g., set to 0.6), so protocol expects CRAs to submit at least 50–80% of the recent average number of CUs in each epoch — otherwise, their rewards may be reduced or blocked
  • β_max ∈ [1.2, 2.0] (e.g., set to 1.5), so the protocol allows to exceed on max 200%
# [Global CU Submission Thresholds]
EMA_n = λ * U_n + (1 - λ) * EMA_n-1
# λ     = smoothing factor (e.g., 0.4)
# U_n   = total CU submissions in current epoch

T_min = β_min * EMA_n
T_max = β_max * EMA_n
# β_min ∈ [0.5, 0.8] → min submission threshold
# β_max ∈ [1.2, 2.0] → max submission threshold

The protocol will:

  • Reject or withhold rewards for submissions if U_n < T_min (to prevent under-reporting).
  • Cap reward eligibility if U_n > T_max (to prevent spamming).

These thresholds are computed on-chain per epoch during the reward pool initialization phase using standard EMA logic:

EMA_n = λ · U_n + (1 − λ) · EMA_{n−1}

Where λ is the smoothing factor (e.g., 0.3–0.5)

U_n: actual CU submissions in the current epoch

1-4 Epochs:

Epoch EMA Source T_min / T_max Enforcement Mode
1 EMA_1 = U_1 None Not enforced
2–4 Simple average Computed Alert-only / soft flag
≥5 True EMA Fully dynamic Full enforcement

Alert Notice for Anomalous CRA Behavior

To further enhance network resilience, the protocol includes anomaly detection alerts based on submission volume trends per agent.

If a CRA's CU submission count deviates significantly (above or below a defined variance threshold) from its 5-day EMA baseline, an alert flag is raised for network monitoring.

Let:

  • s_i(t): CU submissions from CRA i on day t
  • μ_i: 5-day EMA of CRA i's historical submissions
  • δ: Sensitivity threshold for flagging anomalies (e.g. 0.4 means ±40%)
  • Alert condition (= an alert raised if): |s_i(t) − μ_i| / μ_i > δ
# [CRA Anomaly Detection]
Alert_if: |s_i(t) - μ_i| / μ_i > δ
# s_i(t) = CRA i's daily CU submission count
# μ_i    = 5-day EMA of CRA i's submissions
# δ      = anomaly threshold (e.g., 0.4 for ±40%)

This means:

  • If the CRA's daily submission count deviates by more than δ·100% from its average, it's considered anomalous
  • Symmetric: It flags both unusually high or low activity

Example:

  • μ_i = 1,000 (CRA normally submits ~1,000 CUs/day)
  • δ = 0.4 (40% threshold)
  • Alert is raised if daily count is less than 600 or more than 1,400

Triggering an alert:

  • Does not block submissions, but marks them for review. CRA can be notified by warning NFT or simple transaction.
  • Can optionally be linked to slashing, cool-down, or bonus ineligibility based on governance policy.

These flags are:

  • Logged on-chain.
  • Available via explorer or monitoring API.
  • Evaluated at the end of each epoch alongside CU validation.
  • While alerts don't alter P_n, they complement thresholds by flagging rate-of-change, not just absolute numbers.
  • Also used for future stake weighting, or CRA slashing logic.

CRA Bonding Deposit Formula

To prevent manipulation in the consumption reflection process, a self-bonding mechanism is defined for the OUTBE network. CRA bonding is applied in defined number of days (for example 100) after the CRA begins submitting Consumption Units (CUs) to the network. The exact period will be defined in subsequent period.

Bond_i = B_base + B_scale · Û_i
  • Bond_i: Required deposit for CRA i
  • B_base: Minimum fixed base bond (e.g., 1,000 coen)
  • B_scale: How much extra bond is required per CU
  • Û_i: CRA's expected or declared CU submission volume per epoch (or average over past epochs)
# [CRA Bonding Formula]
Bond_i = B_base + B_scale * Ũ_i
# B_base = fixed minimum bond (e.g., 1,000 coen)
# Ũ_i    = declared or average CU volume per epoch

B_scale = (EpochsUntilUnbonding * Ũ_i) / MaxSlashingRisk
# EpochsUntilUnbonding = bonding lock period (e.g., 3 epochs)
# MaxSlashingRisk      = worst-case loss the protocol must cover
B_scale = (EpochsUntilUnbonding × Û_i) / MaxSlashingRisk

Where:

  • MaxSlashingRisk = estimated worst-case damage (e.g., 1–2 full reward epochs)
  • Unbonding period = how long stake is locked (e.g., 3 epochs)

This ensures the bond can cover at least one round of fraudulent over-reporting or abuse.

So if a CRA commits fraud (e.g., submitting duplicate CUs), their bond is slashed.

Initial CRA Offer Logic:

1. Starting Price: 0.001 USDT (coen) per CU

2. Initial Offer: Set by protocol on-chain

3. Validation:

  • If CRA submits → protocol accepts → reward confirmed
  • If no submission → reward offer adjusted downward

4. Reward Multiplier Example:

  • Target reward: 0.06 USDT
  • GM: 20% → base = 0.05
  • Implied multiplier: 0.06 / 0.001 = 60x

Main incentive for CRAs is not reward. It's the value they bring to their customers by enabling consumption reflection.

4. Emission Cap Rationale and Enforcement

To prevent inflationary risks:

P_n ≤ θ · MaxEmission_n 
# [Emission Cap Constraint]
P_n ≤ θ * MaxEmission_n
# Prevents inflation beyond configured policy

This ensures token supply growth aligns with protocol-defined sustainable economic policy. Note that Emission cap will be defined in a subsequent phase and calibrated based on network maturity and ecosystem needs.

5. Comprehensive Workflow Example (Step-by-Step)

1. Epoch Start

  • Oracle and usage data acquisition.
  • Compute initial reward pool using defined formulas.

2. Active Submission Period

  • CRAs aggregate, hash, and submit CUs daily.
  • Protocol transparently logs submission data.

3. Epoch End and Validation

  • Validate CUs for uniqueness and authenticity.
  • Calculate and distribute rewards proportionally to submission count.

4. Preparation for Next Epoch

  • Update EMA-based participation metrics.
  • Oracle prices refreshed.
  • Reward pool recalculated based on evolving metrics.

6. Economic and Operational Implications

  • CRA Predictability: Consistent fiat-targeted rewards mitigate market volatility risks, enhancing operational sustainability.
  • Protocol Resilience: Flexible adjustments protect from sudden participation drops or token market shocks.
  • Community Trust: Transparency and robust anti-manipulation safeguards foster a trusted protocol ecosystem.

The detailed CRA reward mechanism proposed here delivers a sustainable, equitable, and adaptive economic model. By closely tying real-world CRA operational expenses to a dynamic crypto-based reward framework, it ensures continuous participation, predictable rewards, and robust economic integrity. This adaptive mechanism serves as the economic backbone for stable, long-term protocol engagement.

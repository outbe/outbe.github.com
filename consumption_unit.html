CRA: Consumption Unit Generation

June 2025

1. Abstract

Consumption Unit (CU) is a data structure that represents a bundle of a user's Acts of Consumption for a given worldwide day, user's bank account, and time zone. Each CU is generated by the Consumption Reflection Agent (CRA) through the aggregation of eligible transaction records (CRs). Each CR is a reflection of a particular Act of Consumption. It is verified, submitted to the L2 network, and signed by the CRA. After being recorded on L2, CU becomes immutable and serves as the basis for submitting a Tribute to the network.

After the Tribute is formed, it is offered on L1 by User via Wallet App.

Glossary

Term Definition
Coen is a digital asset on Network whose value directly reflects Users' economic activities. Native Coins are minted strictly according to Emission Limit rules and are tradable freely on Exchanges.
Consent ID (consent_id) Unique identifier for a user's data processing consent, required for CR and CU processing.

Consumption Reflection Agent (CRA) is a regulated entity responsible for retrieving, verifying, and posting transaction data. CRAs generate Consumption Units by aggregating a Consumption Records and confirming their legitimacy. They act as the bridge between traditional financial systems and the network's on-chain value creation.
Consumption Unit (CU) is an entity representing a batch of Consumption Records aggregated daily per bank account. Each Consumption Unit qualifies as a whole for Tribute issuance and serves as the basis for Tribute allocation.

Merchant Category Code (MCC) ISO code that categorises merchant types, used to determine eligibility for CU inclusion.

Worldwide Day is a 48-hour period that spans all global time zones, from UTC+14 to UTC−12. It is used as the standard temporal unit for synchronising Epochs, network operations, and global activity across the Network.

2. System Logic

Transactions

The CRA retrieves raw financial transaction data from a user's bank account from the valid financial data sources (e.g.Open Banking APIs or direct integration with financial institutions).

Steps:

  • Authenticate and authorize access to a user's bank account via the bank.
  • Use APIs to pull transaction data.
  • Filter out relevant payment transactions.
  • Normalize data: unify formats across consumption data providers.

Consumption Records (CRs)

The CRA ingests pre-validated transaction data twice per worldwide day.

The first ingestion is initiated after the end of the CRA's local day and covers the period from the beginning of the worldwide day (00.00 UTC+12) up to the end of the CRA's local day.

The second ingestion is initiated after the worldwide day ends and covers the remaining period, from 00:00 of the CRA's local next day to the end of the worldwide day (23:59 UTC–12). Examples: Appendix E.

CRA:

  • Filters transactions by applying eligibility rules (personal accounts, merchant-type).
  • Creates a Consumption Record (CR) in accordance with the Consumption Record Schema and normalizes transaction data.
  • Transforms the value of the Act of Consumption into both the Settlement Token and Coen, by applying the exchange rate from the transaction's original currency at the moment the transaction occurred: initiated_at.
  • Flags refunds as negative-value CRs.
  • Hashes and stores all CRs with traceable, auditable attributes.

Each CR links to:

  • A user's bank account and consent ID. In the Wallet App, the user provides consent via their bank, and the CRA stores this consent and begins fetching transaction data. The CRA is responsible for monitoring consent validity and managing consent flows according to local standards, using commonly adopted statuses such as active, revoked, expired, and pending.
  • The currency exchange rate and its source.
  • Settlement Token and Coen rates. The Settlement Token is defined by the user during account creation in Wallet App. Settlement Token can be changed by the user, but it will become effective after the next day of consumption. The ongoing day will still use previously assigned settlement token. The Coen rate will be determined by the on-chain price oracle.
  • A cryptographic hash of CR.

Consumption Units (CUs)

CUs are bundles of CRs aggregated per a user's bank account, worldwide day, and consumption timezone offset from UTC.

Each CU:

  • Created in accordance with suggested Consumption Unit Schema (Appendix B).
  • Computes the total value in both the Settlement Token and the Coen.

CUs are immutable post-signing and subject to network deduplication.

CUs are aggregated per network account and worldwide day to form a Tribute Draft.

Tribute Draft

In Order to ensure proof of consumption and CU for the original Tribute submitted to L1, the Tribute Draft should be created on L2 for consumption data integrity. Tribute Draft ID on L2 will be computed as a hash of selected internal fields for following validation of original Tribute on L1.

3.CU Aggregation Rules

Rule Description
Aggregation CRs are aggregated to CU

- per user's bank account
- per consumption timezone.
- per worldwide day
Account type Personal Account
Eligibility Only merchant transactions, no intra-user transfers
Refunds Tracked as negative CRs
Signing Flow CRA signs CUs, CUs are aggregated to Tribute Draft, which is submitted to L2 → User gets its consumption data in the form of original Tribute (which appears based on Tribute Draft data) in the Wallet App, signs it via Wallet App and sends to L1
Deduplication No overlapping CUs/CRs per account/date, unique CR and CU required

CR aggregation is collecting multiple CRs (Consumption Records) and combining them into CUs (Consumption Units) that represents the total value of consumption for a specific worldwide day, bank account and time zone.

It includes:

  • Grouping all CRs matching the same bank account, time zone, and worldwide day
  • Summing values (e.g. total settlement_amount_minor, total nominal_qty_minor)
  • Producing one signed CU object from that set

CRA Implementation Principles

  • CUs must be generated from finalized transactions.
  • Refund logic must reconcile past CRs by recording a corresponding negative CR in a subsequent CU, with a reference to the original CU and its worldwide day.
  • CRA must be able to recognize and reflect anomalous activity and fraud patterns as part of its processing logic.
  • CRs creation and CUs aggregation are performed by CRA on the L2 network.
  • Each CU, once signed by CRA, is pushed to Wallet App in a form of Tribute.
  • User reviews and signs Tribute. By signing, the User submits the Tribute to L1.

Hashing

Suggested hashing algorithm is Blake3.

Each CR has unique hash derived from the following attributes.

Suggested CR Hash Inputs

Field Description Included in Hash
initiated_at when the financial institution registered the transaction ✅ Yes
bank_account The user's account details hash (blake3(bic + iban || bban) ✅ Yes
TRN Transaction Reference Number, a unique identifier assigned to a specific financial transaction (will be taken from transaction_id) ✅ Yes

Each CU should have unique hash derived from the following attributes.

Suggested CU Hash Inputs

Field Description Included in Hash
created_at date of CU creation
worldwide_date A 48-hour period covering all global time zones ✅ Yes
consumption_timezone Timezone in which the consumption was registered ✅ Yes
bank_account The user's account details hash (blake3(bic + iban || bban) ✅ Yes
list of CR.hashes Ordered list of hashes of the Consumption Records (CRs) that are being associated with the CU

Each Tribute Draft should have unique hash derived from the following attributes.

Suggested Tribute Draft Hash Inputs

Field Description Included in Hash

Appendix A Status Legend

CU is submitted on L2, checked for duplication. If CU is accepted by network it will be included in the Tribute Draft for the period of worldwide day reflecting consolidated user's consumption.

The event of CU validation and creation on L2 is considered as CU is accepted on the network and is ready for further consolidation in Tribute Draft.

The event of Tribute Draft creation and validation on L2 is considered as consolidated record of user's consumption ready to be signed and submitted as original Tribute on L1.

CR will be accepted on L2 in hash form with the fields stated in Hashing Section.

consumption_record:
  type: object
  description: Represents a payment transaction between merchant and user according to PSD2 standards
  required:
    - transaction_id
    - owner

    - initiated_at

    - location
    - transaction_type
    - transaction_amount

  properties:
      
    transaction_id:
      type: string
      description: Unique identifier for the transaction reference number from user's bank account, TRN
      example: "TX1234567890"

    owner:
      type: string
      description: Network address associated with the owner
      example: 0x1234abcd5678ef901234abcd5678ef901234abcd
    
    bank_account:
		type: string
		description: The user's account details hash (blake3(bic + iban || bban)
		example: a1b2c3d4e5f6789012345678901234567890abcdef1234567890abcdef123456 
    
    initiated_at:
      	type: string
      	format: date-time
      	description: Date and time when the transaction was initiated
      	example: "2025-05-03T14:30:45Z"
      
	consumption_timezone:
  		type: string
  		description: Time zone offset from UTC in ±HH:MM format at the time the consumption took place.
  		example: "+01:00"
      
    settlement_token:
		type: string
      	description: Token used for settlement 
      	example: "USD", "EUR", "GBP", "CNY", "JPY", "HKD"

	settlement_base_amount:
  		type: integer
 		description:  Amount of the Consumption Record in the Settlement Token. Integer part, expressed in natural units (e.g., 10 = 10 EUR). Calculated as  transaction original value / settlement rate. The settlement rate should come from a trusted oracle (e.g., Chainlink).
 		example: 10
 		
	settlement_atto_amount: 
 		type: integer
 		description: Amount of the Tribute in Settlement Tokens. Fractional part, in atto points (e.g. 10^-18) 
 		example: 300000000000000000 # 0.3 * 10^18 
  		
  	settlement_rate:
		type: number
		format: decimal
		description: Conversion rate between transaction_amount and settlement amount (base + atto).

      	type: integer
      	description: Exchange rate used to convert the settlement amount (base + atto) into COEN units. This rate is sourced from a price oracle.
    
    nominal_qty_minor:
    	type: integer
  		description: Value of the Consumption Record in Coen. nominal_qty_minor = settlement_base_amount.settlement_atto_amount / nominal_rate_minor	
      
    status:
      type: string
      enum:
        - COMPLETED
        - PENDING
        - DECLINED
        - REFUNDED
      description: Current status of the transaction
      example: "COMPLETED"
          
    settlement_date:
      	type: string
      	format: date-time
      	description: Date and time when the transaction was settled
      	example: "2025-05-04T10:15:22Z"
    
    merchant_name:
      type: string
      description: Name of the merchant
      example: "City Supermarket"
    
    merchant_id:
      type: string
      description: Unique identifier for the merchant
      example: "MERCH-ID-9876"
    
    merchant_category_code:
      type: string
      description: Merchant category code as defined by card networks (MCC)
      example: "5411"
    
    payment_method:
      type: object
      description: Payment method used for the transaction
      required:
        - type
      properties:
        type:
          type: string
          enum:
            - CARD
            - BANK_TRANSFER
            - WALLET APP
            - OTHER
          description: Type of payment method used
          example: "CARD"
        masked_pan:
          type: string
          description: Masked card number (for card payments)
          example: "****-****-****-1234"
        card_type:
          type: string
          description: Type of card used (for card payments)
          example: "DEBIT"
        wallet_app_provider:
          type: string
          description: Provider of the digital wallet app (for wallet app payments)
          example: "APPLE_PAY"
    
    reference:
      type: string
      description: Reference information for the transaction
      example: "POS-PURCHASE-12345"
    
    location:
      type: object
      description: Location where the transaction took place
      properties:
        city:
          type: string
          description: City where the transaction occurred
          example: "Frankfurt"
        country:
          type: string
          description: Country code where the transaction occurred
          example: "DE"
    
    transaction_type:
      type: string
      enum:
        - PURCHASE
        - REFUND
        - AUTHORIZATION
        - CAPTURE
        - VOID
        - PRE_AUTHORIZATION
        - RECURRING
        - INSTALLMENT
        - CASH_ADVANCE
        - BALANCE_TRANSFER
        - FEE
      description: >
        Type of the transaction that indicates its purpose and processing flow:
        * PURCHASE - Standard purchase of goods or services
        * REFUND - Return of funds to the user for a previous transaction
        * AUTHORIZATION - Initial hold of funds without actual transfer (pending transaction)
        * CAPTURE - Completion of a previously authorized transaction
        * VOID - Cancellation of a transaction before settlement
        * PRE_AUTHORIZATION - Reserved funds for future transactions (like hotel or car rental)
        * RECURRING - Regular payment based on an agreement (subscription)
        * INSTALLMENT - Part of a series of scheduled payments for a single purchase
        * CASH_ADVANCE - Cash withdrawal using a payment card
        * BALANCE_TRANSFER - Moving funds between accounts
        * FEE - Service charge from merchant or payment provider
      example: "PURCHASE"
    
    transaction_amount:
      type: object
      required:
        - amount
        - currency
      properties:
		transaction_amount_base:
      		type: integer
      		description: Integer part of the transaction amount, in natural currency units (e.g., 123 = 123 INR). Represents the whole-number portion.
      		example: 123
		transaction_amount_atto:
      		type: string
      		description: Fractional part of the transaction amount, expressed in atto-units (10^-18 of the base currency). Used for precision beyond standard decimals.
      		example: "450000000000000000"  # 0.45 * 10^18
        currency:
          type: string
          pattern: "^[A-Z]{3}$"
          description: ISO 4217 Alpha 3 currency code
          example: "INR"

Appendix C — Timeline

Timeline: CU and Tribute

T₀ – Start of Worldwide Day (00:00 UTC+12)

  • The CRA begins monitoring for consumption events globally either in one local zone or multiple depending on their financial organisation zones coverage.
  • Each User has its own local 24h period of consumption within worldwide day.
  • All transactions during this Worldwide Day are considered for CR/CU processing by CRA.
  • CRA awaits two ingestion periods split by local time of consumption zone within the one worldwide day.

T₁ – CRA First Ingestion Window (after CRA's local day ends)

  • Time: After local day ends (23:59 local time), note that the current worldwide day is still ongoing, only local 24h window came to an end.
  • Period of consumption: Transactions from 00:00 UTC+12 to the end of CRA's local day.
  • Actions:
    • CRA pulls transaction data.
    • Filters and normalizes relevant merchant payments.
    • Creates CRs per schema
    • Stores CRs with exchange rates, merchant data, and references.
    • Generates CUs, submit to L2, signs CU

T₂ – CRA Second Ingestion Window (after Worldwide Day ends, 23:59 UTC–12)

  • Time: After worldwide day ends: starting at 00:00 of the next Worldwide Day
  • Period of consumption: Transactions from local new day start, 00:00 local time, within the same Worldwide day up to 23:59 UTC–12.
  • Actions:
    • CRA fetches remaining transactions eligible for the same Worldwide Day (in case user is changing local zone, due to a travel).
    • Completes the CR set for the day.
    • Generates the rest CUs, submit to L2, signs CU

T₃ – CU Aggregation and Tribute send to Wallet App (~shortly after T₂)

  • Time: Within ~1h hours after T₂ (depends on CRA implementation).
  • Actions:
    • CUs aggregated into Tribute Draft on L2
    • Wallet App fetches draft tributes from L2

T₄ – Tribute Signing Window Opens

  • Time: Immediately after aggregated CUs in a form of Tribute pushed to the Wallet.
  • Window: 24 hours from the end of the Worldwide Day
  • Actions:
    • Wallet notifies user to review and sign the Tribute.
    • Tribute includes one or more CU(s) for the user's Worldwide Day.
    • User can accept or ignore.
    • If ignored for 24h → Tribute is considered expired though it will never be created on the network and so never recorded.

T₅ – Tribute Submission to L1, Tribute Draft to L2 Submission

  • Time: As soon as the user signs the Tribute.
  • Window: Immediately after user sign and submit to L1
  • Actions:
    • Wallet submits the signed Tribute (CU bundle) to L1.

T₆ – On-chain Validation

  • Time: Subject to L1 Tribute validation.
  • Actions:
    • L1 either accepts the Tribute (verification with Tribute Draft check).
    • If accepted → original Tribute status becomes accepted or muted
    • Tribute Draft status on L2 is changed to utilized

If Tribute is not accepted on L1, it is not created and there is no record on L1 for it.

Appendix D — Consumption & Tribute Timing Matrix relative to Worldwide Day

This scheme provides a time zone–adjusted matrix that illustrates the alignment between consumption events and tribute submission deadlines in relation to Worldwide Day. The period from May 20 to May 23, 2025, is used as an example.

Key Components:

  • Top Row (G/R 48h): Represents the 48-hour window used to determine the timeline for the Worldwide Days.
  • HR Rows (+12:00, +04:00, +00:00, -12:00): Display how consumption periods and tribute submission deadlines are adjusted across multiple global time zones.
  • Consumption Blocks: Mark the exact dates and times within each time zone during which consumption activity qualifies for the corresponding Worldwide day.
  • Associated Actions: Each consumption block is tied to:
    1. The Worldwide Day it contributes to (e.g., 20.05), and
    2. The tribute submission deadline to L1 for that Worldwide Day.
